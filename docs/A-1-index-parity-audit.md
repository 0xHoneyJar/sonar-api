# A-1 Index Parity Audit (Mibera blue belt scope)

> **Sprint**: A-1 (sonar-ponder-migration-v1) · **Task**: T-A1.8 (IMP-006 HIGH_CONSENSUS)
> **Goal**: every envio `public.*` index for Mibera-belt-scoped entities is mirrored
> in `ponder.schema.ts` before any production cutover. Without parity, the
> ponder.* schema will exhibit prod-scale slowness on consumer queries that
> envio's GraphQL surface served fast.

## Methodology

Envio's index generation contract (verified against
`envio@3.0.0-alpha.17`, `generated/node_modules/.pnpm/envio@*/node_modules/envio/src/PgStorage.res`):

```ocaml
let makeCreateIndexQuery = (~tableName, ~indexFields, ~pgSchema) => {
  let indexName = tableName ++ "_" ++ indexFields->Js.Array2.joinWith("_")
  let index = indexFields->Belt.Array.map(idx => `"${idx}"`)->Js.Array2.joinWith(", ")
  `CREATE INDEX IF NOT EXISTS "${indexName}" ON "${pgSchema}"."${tableName}"(${index});`
}
```

Source of truth for Mibera-belt entities is `schema.graphql` filtered to
the 34 entities the Mibera-belt handler writes (`config.mibera.yaml`
contracts ∘ `src/belts/mibera/EventHandlers.mibera.ts` ∘ shared
`src/handlers/*.ts` reused by the belt). Implicit PK indexes are
present on every `id` column and are NOT enumerated below — they're
automatic in both envio and Drizzle/Ponder.

Mibera-belt entity surface (34 entities, derived from

## Ponder schema mirror requirements

Every envio index above MUST be present in `ponder.schema.ts` via the
Drizzle `index()` helper. Ponder/Drizzle naming convention:
auto-generated by Drizzle as `<table>_<column>_index` unless an
explicit name is passed; we keep names implicit (matching envio's
`<table>_<column>` shape closely enough that hasura introspection
parity holds — names differ but the column coverage is identical,
which is what query planners care about).

## Verification

Post-deploy in A-3 staging dry-run (T-A3.2), run:

```sql
-- envio source-of-truth (against pre-cutover Postgres on belt)
SELECT
  schemaname,
  tablename,
  indexname,
  indexdef
FROM pg_indexes
WHERE schemaname = 'public'
  AND tablename IN (
    'MiberaLoan','MintActivity','TrackedHolder','TrackedTokenBalance'
  )
ORDER BY tablename, indexname;

-- ponder mirror (against staging belt post-cold-sync)
SELECT
  schemaname,
  tablename,
  indexname,
  indexdef
FROM pg_indexes
WHERE schemaname = 'ponder'
  AND tablename IN (
    'mibera_loan','mint_activity','tracked_holder','tracked_token_balance'
  )
ORDER BY tablename, indexname;
```

Each envio (table, column-set) pair MUST appear in the ponder result.
Index *names* will differ (Drizzle's auto-naming differs from envio's
`<Table>_<col>` pattern); the load-bearing parity is **column
coverage**, not literal name match. Hasura query planner uses indexes
by column coverage, not name.
