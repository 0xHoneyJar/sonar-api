# CR-203 physical preparation substrate

Physical work is public deployment evidence. Its identity is:

`deployment_id + ownership_index.v1 + operation-scoped capability_version`

Orders, users, contact addresses, and communities are not job ownership.
Legacy `order_id` is retained only as a correlation.

Migration order is mandatory:

1. `001_expand_physical_job_identity.sql`
2. deploy the transactional dual writer everywhere
3. `001b_enable_dual_write.sql`
4. run `kitchen-backfill-job-identity.ts` until it is a no-op
5. run `kitchen-verify-job-parity.ts`
6. disable all legacy writers
7. `002_new_key_authority.sql`
8. `003_constrain_physical_job_identity.sql`

Any parity divergence is persisted and disables admission and worker claims.
An absent migration-authority row also fails closed.
Runtime expansion seeds the initial legacy authority only when the migration
state table is genuinely new; it never repairs a deliberately missing row.

Production admission requires `KITCHEN_PREPARATION_PORT=belt_config_batch` plus
an explicit drain strategy so jobs are never accepted into an immortal queue:

| Drain | Env | Behavior |
|---|---|---|
| file | `KITCHEN_BELT_CONFIG_PATH` | One config rewrite for the whole claimed batch, then optional restart webhook |
| webhook | `KITCHEN_BELT_CONFIG_PATCH_WEBHOOK` | One batched patch-plan POST (10s timeout); webhook owns restart unless `KITCHEN_BELT_WEBHOOK_OWNS_RESTART=false` |
| external_scale | `KITCHEN_PREPARATION_DRAIN=external_scale` (alias: `external`) | Config applied out-of-band (SCALE PR / blue-green); jobs stay `queued` until `POST /v2/collection-preparations/ack` with `drain_mode=external_scale` (or Hasura readiness completes unleashed queued jobs) |

Batch admit is `POST /v2/collection-preparations/batch` (1–50 items; cap =
`BATCH_PREPARATION_MAX_ITEMS`). Override worker claim size with
`KITCHEN_BATCH_CLAIM_LIMIT` (still capped at 50). Physical identity remains one
job per deployment; the worker drains a claim set as a single Belt
materialization (one rewrite / one webhook / one restart). `renewLease` extends
`lease_until` only — it does **not** bump `lease_epoch` (epoch advances on claim).

The `local_config` seam remains non-production only. `/health` reports process
liveness; `/ready` reports admission only when the drain port is available.

The legacy v1 collection-status lookup remains pinned to the first admitted
job for a collection key. It is compatibility history, not a current-capability
selector. Canonical/version-aware clients use the v2 physical job identity;
future current-capability lookup requires an explicit capability version input.

Deliberate blockers:

- `local_config` is not a production preparation/deploy port.
- Hasura rows provide positive readiness evidence, but a live zero-holder
  registration/sync marker does not exist yet.
- No generic ERC-1155 worker exists.
- Solana preparation remains unsupported until CR-402.
- PostgreSQL migration/concurrency CI and live inventory remain external gates.
