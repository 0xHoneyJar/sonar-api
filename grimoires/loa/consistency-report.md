## Consistency Score: 8 / 10

A mature, internally consistent codebase. Naming conventions are followed

## Naming Patterns

### Handler Files (consistent)
- Kebab-case, product-scoped: `honey-jar-nfts.ts`, `sf-vaults.ts`, `mibera-premint.ts`.
- Multi-file modules use a sibling folder with `constants.ts` (e.g.
  `tracked-erc20/constants.ts`, `friendtech/constants.ts`).

### Handler Functions (consistent)
- `handle<Domain><Event>` camelCase: `handleHoneyJarTransfer`, `handleSFVaultDeposit`,
  `handlePremintParticipated`. 84 registrations follow this.

### Entities (mostly consistent)
- PascalCase entity names. Two sub-conventions for namespacing:
  - Underscore prefix: `HoneyJar_Transfer`, `HoneyJar_Approval` (HoneyJar contract events)
  - Concatenated prefix: `SFPosition`, `SFVaultStats`, `HenloVaultRound`, `ApdaoAuction`
- **Minor inconsistency**: only the HoneyJar family uses the `Contract_Event`
  underscore form; every other domain concatenates. Not harmful, but a reader
  must know both forms.

### Entity ID Conventions (consistent ‚Äî documented in ENTITY_REFERENCE.md)
- `{txHash}_{logIndex}` ‚Äî immutable events
- `{chainId}_{address}` ‚Äî per-chain user records
- `{chainId}_{address}_{identifier}` ‚Äî compound records
- `{address}` ‚Äî cross-chain aggregates

### `@entity` / `@index` Directives (inconsistent)
- 10 of 93 entity types carry an explicit `@entity` directive (the Mibera and
  Apdao families); the other 83 rely on implicit entity detection.
- `@index` is applied to 31 fields, concentrated on `address` / `collectionKey`.
- **Observation**: The explicit-`@entity` subset appears to be newer code. Not a
  bug ‚Äî Envio supports both ‚Äî but the mixed style suggests an unfinished migration.

## Conflicts / Flagged Items (NOT changed)

### CF-1: `BERACHAIN_TESTNET_ID` is mislabeled (Medium ‚Äî naming hazard)
- `src/handlers/constants.ts:6` ‚Äî `BERACHAIN_TESTNET_ID = 80094`, but 80094 is
  Berachain **mainnet**. The code self-documents this (`BERACHAIN_ID` aliased at
  line 9) but the misleading name still exists and could be imported by mistake.
- **Breaking-change risk**: renaming/removing `BERACHAIN_TESTNET_ID` would break
  any importer. Flagged for `/implement`, not micro-fixed.

### CF-2: Stat-update approximations (Low)
- `milady-collection.ts:56` and `mibera-collection.ts:159` both set
  `uniqueBurners ?? 1` with a TODO. Two handlers carry the same shortcut ‚Äî a
  shared `lib/` helper would unify the (eventual) correct implementation.

### CF-3: Stats handled inline vs. in lib (Low)
- ERC721 holder logic is centralized in `src/lib/erc721-holders.ts`, but several
  collection handlers (`milady`, `mibera`) re-implement burn-stat updates inline
  rather than via the shared lib. Consistency opportunity, not a defect.

## Improvement Opportunities (non-breaking)

1. Unify entity namespacing ‚Äî pick `Contract_Event` or `ContractEvent` and migrate.
2. Complete the `@entity` directive migration across all 93 types or none.
3. Extract a shared burn-stats helper into `src/lib/` (resolves CF-2 + CF-3).
4. Rename `BERACHAIN_TESTNET_ID` ‚Üí `BERACHAIN_ID` and update importers (CF-1 ‚Äî via `/implement`).

## Healthy Patterns Worth Keeping

- Config-as-manifest: zero hardcoded contract addresses in handler logic flow;
  all addresses live in `config.yaml` or `handlers/constants.ts` tables.
- Shared `recordAction` decouples quest verification from per-protocol handlers.
- One module per product keeps blast radius small per change.
